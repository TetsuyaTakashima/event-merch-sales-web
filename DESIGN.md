# イベント物販売上管理Webアプリ 設計メモ

作成日: 2026-05-05

## 方針

この初期実装は、外部依存なしでブラウザから直接開ける静的Webアプリとして作成する。販売登録、在庫反映、販売履歴、集計、CSV出力の主要な業務フローを先に触れる形にし、将来的にバックエンドAPIや認証基盤へ移行しやすい構成にする。

## 初期実装のスコープ

- ダッシュボード
- 販売登録
- 販売履歴と取消
- 在庫確認と在庫調整
- 売上集計
- CSV出力
- データバックアップ
- データ復元
- イベント管理
- イベント編集・削除
- 商品追加
- 商品編集
- ユーザーと権限の簡易管理
- ユーザー編集・削除

## 画面構成

| 画面 | 目的 |
| --- | --- |
| ダッシュボード | イベント当日の売上、販売点数、低在庫、直近販売を確認する |
| 販売 | 商品を選び、カートに入れて決済方法を選択し、販売を確定する |
| 販売履歴 | 販売明細を確認し、権限に応じて取消する |
| 在庫 | 現在庫、初期在庫、実在庫、差異、在庫調整を管理する |
| 集計 | 商品別、決済方法別、時間帯別の売上を確認し、CSV出力する |
| イベント | イベントの追加、編集、削除、状態変更、対象イベント選択を行う |
| 商品 | 商品とバリエーションの一覧確認、商品追加、編集、販売状態変更を行う |
| ユーザー | 操作ユーザーと権限の追加、編集、削除を行う |

## 状態管理

- Supabase設定時はアプリ全体の永続状態を `app_state` に共有保存する。
- Supabase未設定時はアプリ全体の永続状態を `localStorage` に保存する。
- 販売カート、検索条件、選択中タブなどの一時状態はメモリで管理する。
- 販売確定時は、販売データ保存と在庫減算を同一処理内で行う。
- 販売取消時は、販売ステータス更新と在庫戻しを同一処理内で行う。
- Supabaseモードの管理保存は `version` を使った楽観ロックで競合を検知し、競合時は最新状態を再読込する。
- Supabaseモードの販売・取消・在庫操作はRPC内で `app_state` 行をロックして処理し、同時更新時の在庫ズレと権限迂回を抑える。
- Supabaseモードでは `app_state` のRealtime更新を購読し、別端末の更新を画面へ反映する。
- テスト販売ユーザーは `app_state.id = 'main'` ではなく `sandbox:<ユーザーID>` を使用し、本番売上・在庫・イベントと分離する。
- 画面上からJSON形式で全データをバックアップ・復元できる。

## デプロイ方針

静的アプリはNetlify、Vercel、GitHub Pagesなどで公開できる。複数スタッフ運用ではSupabase AuthとDatabaseを設定する。

現在は主要データを1つの共有JSONで管理している。競合による上書きは防止しているが、より強いDB側の権限分離や監査が必要な規模では、売上・売上明細・在庫・商品を個別テーブル化し、販売登録をDB関数のトランザクションへ移行する。
